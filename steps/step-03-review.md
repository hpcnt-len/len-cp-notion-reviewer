## 📚 문서 리뷰 핵심 철학

모든 검토는 아래 3가지 핵심 철학을 기반으로 수행합니다.

### 1. Write to Read (읽는 사람 중심)

> **쓰는 사람이 편한 글이 아니라, 읽는 사람이 편한 글을 쓴다.**

| 검토 관점 | 질문 |
|----------|------|
| **이해 용이성** | 배경 지식이 없는 동료도 이해할 수 있는가? |
| **정보 접근성** | 필요한 정보를 빠르게 찾을 수 있는가? |
| **구조적 명확성** | 문서의 흐름이 논리적이고 예측 가능한가? |

**검토 시 확인:**
- 전문 용어에 대한 설명이 있는가?
- 결론/요약이 먼저 나오고 세부 내용이 뒤따르는가? (Top-down)
- 읽는 사람의 시간을 절약해주는 구조인가?

### 2. Focus on "Why" & "Decision" (결정 중심)

> **단순 사실 나열보다 '왜 이 결정을 했는지', '무엇을 포기했는지(Trade-off)'가 핵심이다.**

| 검토 관점 | 질문 |
|----------|------|
| **결정의 이유** | "왜 이 방식을 선택했는가?"가 명시되어 있는가? |
| **Trade-off** | 포기한 대안과 그 이유가 기록되어 있는가? |
| **맥락 보존** | 미래의 독자가 결정의 배경을 이해할 수 있는가? |

**검토 시 확인:**
- "What"만 있고 "Why"가 없는 섹션이 있는가?
- 여러 대안 중 선택한 이유가 명시되어 있는가?
- 제약 조건이나 고려 사항이 기록되어 있는가?

### 3. Living Document (살아있는 문서)

> **문서는 코드가 배포될 때 함께 업데이트되어야 한다. (Stale 문서 지양)**

| 검토 관점 | 질문 |
|----------|------|
| **현재성** | 문서 내용이 현재 시스템 상태를 반영하는가? |
| **유지보수성** | 업데이트하기 쉬운 구조인가? |
| **버전 정보** | 마지막 업데이트 일시가 명시되어 있는가? |

**검토 시 확인:**
- "예정", "계획" 등 과거의 미래형 표현이 현재도 유효한가?
- 더 이상 사용하지 않는 시스템/기능에 대한 설명이 있는가?
- 코드와 문서의 불일치 가능성이 있는 부분이 있는가?

## REVIEW CRITERIA:

### 공통 검토 기준 (모든 문서 타입)

| 기준 | 검토 내용 |
|------|----------|
| **가독성** | 긴 문단 분리, 불렛 포인트/서식 활용 |
| **Notion 스타일** | H1~H3, 인용구, 콜아웃 활용 |
| **이모지 제한** | 전체 텍스트 대비 5% 이내 |
| **읽는 사람 중심** | 배경 지식 없이도 이해 가능한 구조 |
| **결정 중심** | Why와 Trade-off가 명시되어 있는지 |
| **문서 현재성** | Stale한 내용이 없는지 |
| **표 활용** | 복잡한 구조적 내용에 표 사용 |

### 제안하지 않는 경우

1. 섹션 제목이 다르지만 **동일한 성격의 내용**이 존재
2. IT 기술 조직에서 **합리적인 형태**로 판단되는 경우
3. 가이드라인과 다르지만 **맥락상 적절한 구조**인 경우

| 섹션 | 필수 내용 |
|------|----------|
| **1. Overview** | 구현 목표 한 문장 요약, 관련 문서 링크 |

| **4. Testing Strategy** | TC 핵심 시나리오, Unit Test 커버리지 목표 |

**검토 체크리스트 (반드시 따를 필요 없음, 부족한 내용은 feedback 에서 제안):**
- [ ] 구현 목표가 명확히 정의되어 있는가?
- [ ] DB 스키마 변경 사항이 구체적으로 기술되어 있는가?
- [ ] API Request/Response 명세가 JSON 예시와 함께 있는가?
- [ ] 복잡한 비즈니스 로직이 Flowchart나 Pseudocode로 설명되어 있는가?
- [ ] 기존 기능에 미치는 영향(Side Effects)이 분석되어 있는가?
- [ ] 데이터 마이그레이션 필요 시 전략이 있는가?
- [ ] 테스트 시나리오와 커버리지 목표가 있는가?

#### 4b. 시스템 소개/기획 (System Planning)

> **목적:** 프로젝트를 왜 해야 하는지 설득하고, 어디까지 할 것인지 범위를 확정함.
> **특징:** Amazon의 'Working Backwards'를 경량화.

**권장 구조 (반드시 따를 필요 없음, 부족한 내용 제안용):**

| 섹션 | 필수 내용 |
|------|----------|
| **1. Problem Statement (Why)** | 현재 문제점 (수치/고객 피드백 기반), 해결하지 않았을 때의 비용 |
| **2. Proposed Solution (What)** | 해결책 요약, Success Metrics (정량적 지표) |
| **3. Scope** | In-Scope (반드시 구현할 것), **Non-Goals** ⭐ (하지 않기로 결정한 것) |

**검토 체크리스트 (반드시 따를 필요 없음, 부족한 내용은 feedback 에서 제안):**
- [ ] 문제점이 수치나 고객 피드백 기반으로 설명되어 있는가?
- [ ] 해결하지 않았을 때의 비용/리스크가 명시되어 있는가?
- [ ] Success Metrics가 정량적으로 정의되어 있는가? (예: API 응답속도 20% 개선)
- [ ] **Non-Goals**가 명확히 정의되어 있는가? (범위 확장 방지용 방파제)

#### 4c. 시스템 설계 (System Design)

> **목적:** 아키텍처 구조를 잡고 기술적 의사결정의 근거를 남김.
> **특징:** Google의 Design Doc 스타일 차용. **'Alternatives' 섹션이 가장 중요함.**

**권장 구조 (반드시 따를 필요 없음, 부족한 내용 제안용):**

| 섹션 | 필수 내용 |
|------|----------|
| **1. High-Level Architecture** | 전체 시스템 구성도 (Diagram 필수), 데이터 흐름도 |
| **2. Key Technology Decisions** | 기술 스택 및 선정 이유 |
| **3. Alternatives Considered** ⭐⭐ | Option A (선택): 장단점, Option B (버림): 장단점, **Why A over B?** (트레이드오프 분석) |
| **4. Scalability & Reliability** | 확장 전략 (Scale-out/Scale-up), 장애 복구 전략 (Failover) |
| **5. Security / Privacy** | 개인정보 처리 및 보안 이슈 |

**검토 체크리스트 (반드시 따를 필요 없음, 부족한 내용은 feedback 에서 제안):**
- [ ] 전체 시스템 구성도(Diagram)가 있는가?
- [ ] 데이터 흐름이 명확히 표현되어 있는가?
- [ ] **대안(Alternatives)이 분석되어 있는가?** (가장 중요!)
- [ ] 왜 특정 기술/방식을 선택했는지 Trade-off 분석이 있는가?
- [ ] 트래픽 증가 시 확장 전략이 있는가?
- [ ] 장애 발생 시 복구 전략이 있는가?
- [ ] 보안/개인정보 이슈가 점검되어 있는가?

#### 4e. 업무 단위 티켓 (Action Item Ticket)

> **목적:** 실무자가 '무엇을', '언제까지', '어떤 상태여야 완료인지' 명확히 인지하게 함.
> **특징:** 업무의 **맥락(Context)**과 **완료 조건(AC)**이 핵심.

**권장 구조 (반드시 따를 필요 없음, 부족한 내용 제안용):**

| 섹션 | 필수 내용 |
|------|----------|
| **Title** | [BE/FE] 명확한 작업 명 (예: 유저 프로필 수정 API 구현) |
| **1. Context (Why)** ⭐ | 이 작업이 왜 필요한지 1~2줄, 관련 링크 (있으면) |
| **2. Acceptance Criteria (AC)** ⭐ | 완료 조건 체크리스트 (구체적인 조건들) |

**검토 체크리스트 (반드시 따를 필요 없음, 부족한 내용은 feedback 에서 제안):**
- [ ] Title이 [BE/FE] 등 태그와 함께 명확한가?
- [ ] **Context(Why)**가 충분히 설명되어 있는가? (기계적 코딩 방지)
- [ ] 관련 기획서/디자인 링크가 첨부되어 있는가?
- [ ] **Acceptance Criteria**가 체크리스트 형태로 구체적인가?
- [ ] 각 AC가 검증 가능한 조건인가? (예: "입력값이 유효하지 않을 때 400 에러 리턴")

**TL;DR / 요약 추가 규칙:**
TL;DR 또는 요약 섹션은 **문서가 일반적인 형태를 잘 갖추고 있는 경우에만** 추가를 제안합니다.

| 조건 | TL;DR 제안 여부 |
|------|----------------|
| 문서 구조가 체계적이고 완성도가 높음 | ✅ 제안 |
| 문서가 아직 Draft 단계로 구조가 미흡함 | ❌ 제안하지 않음 |
| 내용이 너무 짧아 요약이 불필요함 | ❌ 제안하지 않음 |
| 이미 도입부에 충분한 요약이 있음 | ❌ 제안하지 않음 |

> 💡 **이유:** 미완성 문서에 TL;DR을 추가하면 본문 변경 시마다 TL;DR도 수정해야 하므로, 문서가 어느 정도 완성된 후에 추가하는 것이 효율적입니다.

**목차(TOC) 검토:**
문서 길이가 **읽는 데 3분 이상** 걸리는 수준인지 판단합니다.

- 대략적 기준: 본문 1,500자 이상 또는 주요 섹션 5개 이상
- 3분 이상 소요 문서에 목차가 없으면 → **목차 추가 필수** 제안

| 문서 길이 | 목차 필요 여부 |
|----------|---------------|
| 짧음 (3분 미만) | 선택 사항 |
| 김 (3분 이상) | **필수** |

> ⚠️ **TOC 추가 제한사항**
> 
> Notion의 Table of Contents 블록은 **Notion MCP API로 직접 생성할 수 없습니다.**
> TOC 추가가 필요한 경우 step-06에서 **수동 추가 안내**를 제공합니다.
> 
> **Notion에서 TOC 추가 방법:** 문서 상단에서 `/toc` 입력 → Table of Contents 선택

### 5. 가독성 및 스타일 검토

- 긴 문단(5줄 이상)이 분리되지 않은 경우
- 불렛 포인트나 서식 부족
- Notion 네이티브 스타일(헤딩, 인용구, 콜아웃) 미활용
- 복잡한 구조적 내용이 줄글로 작성된 경우

### 6. 헤딩 구조 검토 (H1~H3)

문서의 헤딩이 맥락에 맞게 계층적으로 구성되어 있는지 검토합니다.

**6a. 헤딩 계층 구조**
- H1은 문서 내에서 1개만 사용 (문서 제목)
- H2는 주요 섹션 구분용
- H3는 H2 하위의 세부 항목용

**올바른 구조 예시:**
```
# 문서 제목 (H1)
## 배경 (H2)
### 현재 상황 (H3)
### 문제점 (H3)
## 목표 (H2)
## 설계 (H2)
### 아키텍처 (H3)
### 데이터 모델 (H3)
```

**6b. 검토 항목**

| 문제 유형 | 설명 | 개선 방향 |
|----------|------|----------|
| 헤딩 누락 | 긴 섹션에 헤딩 없이 나열 | 논리적 단위로 H2/H3 추가 |
| 계층 건너뛰기 | H1 다음 바로 H3 사용 | H2를 먼저 사용 |
| 과도한 헤딩 | 짧은 내용에 불필요한 헤딩 | 헤딩 제거 또는 통합 |
| 불일치 수준 | 같은 수준 내용에 다른 헤딩 레벨 | 동일 레벨로 통일 |

**6c. 제안하지 않는 경우**
- 문서가 짧아 헤딩 구조가 단순해도 되는 경우
- 의도적으로 평면적 구조를 사용한 경우 (예: 체크리스트)

### 7. 텍스트 강조 검토

문맥상 중요한 내용이 적절히 강조되어 있는지 검토합니다.

**7a. 강조 방식**

| 강조 유형 | 용도 | 예시 |
|----------|------|------|
| **Bold** | 핵심 키워드, 중요 개념 | **필수**, **주의**, **API 엔드포인트** |
| *Italic* | 참고 사항, 부연 설명, 용어 첫 등장 | *단, 이 경우는 제외*, *OAuth 2.0* |
| `Backtick` | 코드, 명령어, 기술 용어, 설정값 | `GET /api/users`, `true`, `config.yaml` |
| ~~취소선~~ | 더 이상 유효하지 않은 내용 | ~~이전 방식~~ |

**7b. 검토 기준**

| 상태 | 설명 |
|------|------|
| ✅ 적절 | 중요 내용이 적절히 강조됨 |
| ⚠️ 부족 | 중요한 키워드/코드가 강조 없이 나열됨 |
| ⚠️ 과다 | 너무 많은 텍스트가 강조되어 효과 감소 |

**7c. 강조가 필요한 경우**
- 기술 용어, API 엔드포인트, 명령어 → `backtick`
- 필수/주의/중요 등 핵심 키워드 → **bold**
- 설정값, 파일명, 변수명 → `backtick`
- 특정 용어의 첫 등장 시 정의 → *italic* 또는 **bold**

**7d. 제안하지 않는 경우**
- 이미 충분히 강조가 적용된 경우
- 코드 블록 내부 (이미 서식 적용됨)
- 인용문 내부

### 8. 이모지 사용량 체크

전체 텍스트 대비 이모지 비율 계산:
- 5% 이하: OK
- 5% 초과: 개선 제안 대상

### 9. 논리적 전개 검토

문서 전체를 읽고 제목과 내용의 논리적 연결성을 평가합니다.

**9a. 제목-내용 일치도**
- 문서 제목이 본문의 핵심 내용을 정확히 반영하는가?
- 제목에서 기대되는 내용이 실제로 다뤄지는가?
- 제목과 전혀 무관한 내용이 주를 이루지는 않는가?

**평가 기준:**
| 상태 | 설명 |
|------|------|
| ✅ | 제목이 본문 내용을 잘 대표함 |
| ⚠️ | 제목과 내용이 부분적으로 불일치 |
| ❌ | 제목이 내용을 전혀 반영하지 못함 |

**9b. 논리적 흐름**
- 섹션 순서가 자연스러운가? (일반적: 배경 → 목표 → 내용 → 결론)
- 앞 섹션에서 언급한 내용이 뒤 섹션에서 적절히 이어지는가?
- 갑자기 맥락 없이 등장하는 내용이 없는가?

**9c. 용어/개념 일관성**
- 동일한 개념에 다른 용어를 혼용하지 않는가?
- 정의 없이 갑자기 등장하는 전문 용어나 약어가 없는가?
- 있다면 첫 등장 시 설명이 필요함을 표시

**9d. 줄임말(약어) 감지 및 검토**

문서 내 줄임말을 감지하고 정의 여부를 검토합니다.

**줄임말 감지 패턴:**
- 영문 대문자 2~5자 연속 (예: ODC, PG, MCP)
- 괄호 안 정의가 있는지 확인 (예: "ODC(Open Data Contract)")

**기술조직 공통 줄임말 (정의 불필요):**
다음 줄임말은 기술조직에서 보편적으로 사용되므로 정의 없이 사용 가능:
```
API, DB, SDK, IDE, UI, UX, URL, HTTP, HTTPS, REST, JSON, XML, 
HTML, CSS, JS, TS, SQL, NoSQL, AWS, GCP, K8S, CI, CD, PR, MR,
QA, UAT, SLA, KPI, OKR, ETL, ML, AI, LLM, TDD, BDD, DDD, MVP,
POC, SaaS, PaaS, IaaS, B2B, B2C, CRUD, JWT, OAuth, SSO, RBAC,
MCP
```

**검토 기준:**
| 상태 | 조건 |
|------|------|
| ✅ 정의됨 | 첫 등장 시 `약어(Full Name)` 형태로 정의됨 |
| ✅ 정의됨 | Glossary/용어집 섹션에 정의됨 |
| ✅ 공통 | 기술조직 공통 줄임말 목록에 포함 |
| ⚠️ 미정의 | 정의 없이 사용됨 → **step-05에서 사용자에게 정의 요청** |

**감지된 줄임말 목록 (미정의만 표시):**
```
| # | 줄임말 | 정의 상태 | 첫 등장 위치 |
|---|--------|----------|-------------|
| 1 | [약어] | ⚠️ 미정의 | [섹션명] |
```

**제안하지 않는 경우:**
- 전체적으로 논리적 흐름이 자연스러운 경우
- 약간의 순서 차이가 있어도 문서 이해에 문제없는 경우
- 해당 문서 타입에서 일반적으로 허용되는 구조인 경우

### 10. 문장 품질 검토 (Draft 문서용)

문서의 각 문장을 읽고 문법, 표현, 명확성을 검토합니다.
**특히 Draft 상태 문서에서 자주 발견되는 문제를 집중 검토합니다.**

**문맥 기반 수정 원칙**
 
비문이나 어색한 표현을 수정할 때 **기계적인 1:1 변환을 하지 않습니다.**
반드시 앞뒤 문맥을 파악하여:
- 해당 문장이 전달하려는 **의도**를 이해하고
- 앞 문장과 뒷 문장의 **흐름**을 고려하여
- 문서 전체의 **톤과 스타일**에 맞게
 
**자연스럽게 읽히는 문장**으로 재작성합니다.

**10a. 비문 (문법 오류)**
- 주어-서술어 호응 오류
- 조사 사용 오류
- 문장 성분 누락

| 유형 | 문제 예시 | 수정 예시 |
|------|----------|----------|
| 조사 오류 | "데이터를 처리를 한다" | "데이터를 처리한다" |
| 호응 오류 | "목표는 성능 개선이다를 한다" | "목표는 성능 개선이다" |
| 성분 누락 | "처리된다" (주어 불명) | "시스템이 데이터를 처리한다" |

**10b. 어색한 표현**
- 번역투 표현
- 불필요한 피동/사동
- 중복 표현

| 유형 | 문제 예시 | 수정 예시 |
|------|----------|----------|
| 번역투 | "~에 대해서 ~을 한다" | "~을 한다" |
| 불필요 피동 | "진행이 되어진다" | "진행된다" |
| 중복 표현 | "다시 재시도", "미리 사전에" | "재시도", "사전에" |

**10c. 구어체 → 문어체 변환**
Draft에서 흔히 발견되는 구어체를 문어체로 변환 제안

| 구어체 | 문어체 |
|--------|--------|
| "이거 해야됨" | "이 작업이 필요함" |
| "~하면 될듯" | "~하면 된다" |
| "일단 ~해보자" | "~을 시도한다" |
| "~인듯?" | "~으로 보인다" |
| "ㅇㅇ", "ㄴㄴ" | 정식 표현으로 변환 |

**10d. 불명확한 지시어**
- "이것", "그것", "저것" 등 지시어가 가리키는 대상이 불명확한 경우
- "위의", "아래의" 등 상대적 위치 참조가 모호한 경우

**10e. 중복 문장 삭제**
- 동일하거나 거의 유사한 의미의 문장이 반복되는 경우 삭제
- 같은 내용을 다르게 표현한 문장이 연속으로 나타나는 경우 하나로 통합

| 유형 | 문제 예시 | 수정 예시 |
|------|----------|----------|
| 완전 중복 | "데이터를 저장한다. 데이터를 저장합니다." | "데이터를 저장한다." |
| 의미 중복 | "성능을 개선한다. 퍼포먼스를 향상시킨다." | "성능을 개선한다." |

**10f. 짧은 문장 통합**
- 짧은 문장이 여러 번 반복되고 문맥상 연결 가능한 경우 자연스럽게 통합
- 단, 의도적으로 강조하기 위해 짧게 쓴 문장은 유지

| 문제 예시 | 수정 예시 |
|----------|----------|
| "API를 호출한다. 결과를 받는다. 결과를 파싱한다." | "API를 호출하여 결과를 받고 파싱한다." |
| "사용자가 입력한다. 시스템이 검증한다. 저장된다." | "사용자가 입력하면 시스템이 검증 후 저장한다." |

**10g. 톤 통일 (존댓말)**
문서 전체의 어미 형태를 **존댓말(`~입니다`, `~합니다`)** 로 통일합니다.

**검토 기준:**
- 문서 내 반말(`~한다`, `~이다`)과 존댓말이 혼용되는지 확인
- 기술 문서의 전문성을 위해 **존댓말 사용** 권장

| 비일관 예시 | 수정 예시 |
|------------|----------|
| "API를 호출한다. 결과가 반환됩니다." | "API를 호출합니다. 결과가 반환됩니다." |
| "다음 단계로 진행함. 완료되면 알림이 전송됩니다." | "다음 단계로 진행합니다. 완료되면 알림이 전송됩니다." |
| "설정을 변경해야 함" | "설정을 변경해야 합니다" |

**톤 변환 패턴:**
| 반말/평서형 | 존댓말 |
|------------|--------|
| ~한다 | ~합니다 |
| ~이다 | ~입니다 |
| ~했다 | ~했습니다 |
| ~된다 | ~됩니다 |
| ~임 / ~함 | ~입니다 / ~합니다 |
| ~해야 함 | ~해야 합니다 |

**제안하지 않는 경우:**
- 코드 블록 내 주석
- 인용문
- 불렛 포인트의 짧은 항목 (예: "설정 완료", "테스트 통과")

**검토 결과 형식:**
```
| # | 위치 | 유형 | 원문 | 수정 제안 |
|---|------|------|------|----------|
| 1 | [섹션명] | 비문 | "원문 문장" | "수정된 문장" |
| 2 | [섹션명] | 구어체 | "원문 문장" | "수정된 문장" |
```

**제안하지 않는 경우:**
- 인용문이나 코드 블록 내 텍스트
- 의도적으로 비격식체를 사용한 경우 (예: 인용구 내 개인 메모)
- 기술 용어나 고유명사

### 11. Context 미완성 시 관련 문서 검색 (업무 단위 티켓)

Context가 충분하지 않은 경우:
- Notion MCP `notion-search` 또는 `notion-query-data-sources` 사용
- 동일 Project Name 문서 우선 검색
- 최근 문서 우선
- 최대 3~5개 추천

### 12. 문서 길이 및 분리 검토

문서가 지나치게 긴 경우, **문서 분리**를 제안합니다.

**분리가 필요한 조건 (하나 이상 해당 시):**

| 조건 | 기준 | 설명 |
|------|------|------|
| **물리적 길이** | 읽는 데 약 7~8분 이상 | 본문 약 3,500자 이상 또는 주요 섹션 10개 이상 |
| **주제 응집성** | 2개 이상의 대주제 혼재 | 예: 시스템 설계와 컨셉 소개가 한 문서에 섞임 |
| **구조적 깊이** | H1→H2→H3 구조가 과도하게 복잡 | H3 이하로 3단계 이상 들어가거나, 전체 헤딩 수 15개 이상 |

**12a. 물리적 길이 판단**

문서의 총 글자 수와 섹션 수를 기준으로 판단:
- 본문 약 3,500자 이상 → 읽는 데 7~8분 소요 추정
- 주요 섹션(H1, H2) 10개 이상

**12b. 주제 응집성 판단**

문서 전체를 읽고 서로 다른 대주제가 혼재되어 있는지 분석:
- 제목에서 기대되는 주제와 다른 주제가 상당 부분을 차지하는가?
- 독립적인 문서로 분리해도 각각 의미 있는 문서가 되는가?

| 혼재 유형 | 예시 |
|----------|------|
| 설계 + 소개 | "시스템 A 설계" 문서에 "시스템 A란?" 소개가 절반 이상 |
| 구현 + 기획 | "기능 B 기획" 문서에 상세 구현 스펙이 포함 |
| 여러 시스템 | "시스템 A, B, C 비교" 문서가 각 시스템 상세 설명으로 확장 |

**12c. 구조적 깊이 판단**

헤딩 구조가 과도하게 복잡한지 분석:
- H3 이하(H4 수준)로 3단계 이상 들어가는 섹션이 있는가?
- 전체 헤딩(H1+H2+H3) 수가 15개 이상인가?

**12d. 분리 제안 결과**

분리가 필요한 경우 다음 형식으로 정리:

```
**📄 문서 분리 검토 결과**

| 조건 | 상태 | 비고 |
|------|------|------|
| 물리적 길이 | ⚠️ 분리 권장 | 약 [N]자, 읽는 데 [N]분 추정 |
| 주제 응집성 | ⚠️ 분리 권장 | [대주제1], [대주제2] 혼재 |
| 구조적 깊이 | ✅ OK / ⚠️ 분리 권장 | 헤딩 [N]개, 최대 깊이 [N]단계 |

**분리 제안:**
| # | 분리할 주제 | 제안 제목 | 포함 섹션 |
|---|------------|----------|----------|
| 1 | [주제1] | "[제안 제목]" | [섹션 목록] |
| 2 | [주제2] | "[제안 제목]" | [섹션 목록] |
```

**분리하지 않는 경우:**
- 의도적으로 종합 문서로 작성된 경우 (예: "시스템 A 종합 가이드")
- 분리 시 문맥이 끊어져 이해가 어려워지는 경우
- 문서 길이가 길지만 주제가 일관된 경우
